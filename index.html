Microsoft Foundry / What's New
Ignite 2025 · Jan 2026 rename

Microsoft Foundry — the evolution of Azure AI Foundry

Announced at Ignite 2025 on 18 November 2025 and formalised in the January 2026 Product Terms, Azure AI Foundry is now Microsoft Foundry. Dropping "Azure" signals that agents are first-class citizens across the entire Microsoft ecosystem — not just another Azure service. This tab explains what's new under the hood, beyond the rebrand.

◔
◑
◒
MS FOUNDRY
Rename

Azure AI Foundry → Microsoft Foundry

Formalised 1 January 2026 in the Microsoft Product Terms. Azure AI Foundry Models are now "Microsoft Foundry Models (includes Azure OpenAI)". No functional break — all existing workloads continue to run.

Multi-model

Anthropic Claude, plus OpenAI, Meta, Cohere, Mistral

Foundry now serves models from non-OpenAI providers natively, with over 11,000 models in the catalogue. Foundry's pitch is model-agnostic: pick the best model for each task without switching platforms.

Foundry IQ

Foundry IQ — unified enterprise grounding

A single retrieval layer that connects agents to SharePoint, OneDrive, Fabric, Teams chat, Microsoft Graph and third-party sources. Removes the need to hand-wire each knowledge source per agent.

Agent Service

Foundry Agent Service (GA)

Managed multi-agent orchestration with BYOS, private networking, and Entra-scoped identity per agent. LangGraph and Semantic Kernel both plug in. Replaces custom orchestration code for most teams.

Quality

Task Adherence (preview)

Runtime evaluator that scores whether an agent actually completed the task it was asked to do — catches "sounds right but wrong" outcomes. Integrates with the Agent Service for automatic flagging.

Governance

Entra Agent ID + global admin governance

Every agent is a first-class Entra identity. Global admins can enumerate, audit and revoke agents from a single pane — critical for organisations nervous about agent sprawl.

Fine-tuning

Azure Developer CLI fine-tuning extension

Fine-tune models directly from the terminal without leaving your repo. Versioned runs, evaluation, and promotion to Foundry endpoints scripted end-to-end.

Tools

Computer-use, image-gen, SharePoint & Fabric tools

Built-in agent tools for navigating desktop UIs, generating images, querying SharePoint and Fabric data agents — each in managed preview and callable via function-calling.

Local

Foundry Local SDK

Run Foundry agents and models on-device or on-prem for sovereignty, air-gapped or low-latency scenarios. Legacy SDK still supported; a migration guide ships with the new SDK.

Learn & Explain / CAF AI Agent Decision Tree
Microsoft Cloud Adoption Framework · updated Dec 2025

AI Agent Decision Tree

The CAF decision tree walks you from "I have a potential agent use case" through two parallel tracks — Business Plan (should you build an agent at all?) and Technology Plan (if yes, what do you build it on?). The animation below walks the path a typical enterprise takes when evaluating an agent use case.

How to use this tree

The tree has two parallel tracks. The Business Plan side asks whether you even need an agent — many use cases are better served by deterministic code, RAG, or a SaaS Copilot. Only when reasoning and tool orchestration are genuinely required does the Technology Plan side become relevant.

  • Structured task? Use code — faster, cheaper, more reliable.
  • Static knowledge retrieval? Use classic RAG — no agent needed.
  • SaaS agent fits? Adopt the prebuilt one (M365 Copilot, Dynamics 365, Security Copilot).
  • None of the above? Build custom on Foundry, Copilot Studio or custom infra.

Picking your build platform

  • Copilot Studio — low-code. Best for business teams building internal productivity agents.
  • Microsoft Foundry — pro-code, multi-model, multi-agent orchestration. Best for engineering teams building differentiated customer-facing agents.
  • Custom infrastructure — AKS + OSS frameworks (LangGraph, Semantic Kernel, your own). Best when Foundry's opinions don't fit — niche compliance, bespoke model-hosting, experimental research.

Pattern to remember: start on the lowest-effort path that satisfies the need. Only move to custom infra when you've proven the SaaS and Foundry paths can't deliver.

Learn & Explain / Agentic AI
Concept · Agentic AI

Agentic AI — Perceive, Reason, Plan, Act, Observe

An agent is an AI program that doesn't just respond — it decides. It perceives its environment, reasons about the goal, plans a sequence of tool calls, executes them, and observes the result — then loops. The animation below shows that loop running around a central reasoning model.

The 5 phases of the agent loop

  • Perceive — ingest the user message, environment state, prior memory, and any system events.
  • Reason — the LLM interprets the goal against what it knows (memory, retrieved context, tool schemas).
  • Plan — decompose into subtasks; pick which tool to call next and with what arguments.
  • Act — execute the chosen tool (API call, code execution, DB query, another agent).
  • Observe — read the tool result; update memory; decide whether the goal is met or another loop is needed.

Why agents ≠ chatbots

A chatbot answers. An agent pursues. That distinction matters because it changes every part of how you build, secure, and operate the system.

  • Chatbots are stateless Q&A engines. No tools, no side-effects, no memory beyond the conversation.
  • Agents have goals, tools, memory and autonomy to decide what to do next — which means they can take real actions (send email, move money, write files) and cause real damage if misaligned.
  • Consequence — every agent needs a charter, a bounded tool list, approval gates for destructive actions, and an audit trail. OWASP's ASI Top 10 is the industry's best list of what to guard against.
Learn & Explain / Model Context Protocol (MCP)
Concept · MCP

MCP — the USB-C of tool integration for LLMs

Model Context Protocol is an open standard (from Anthropic, now adopted broadly) that lets AI agents discover and call tools through a uniform interface. Instead of hand-wiring each tool into each agent, an MCP server exposes tools once and any MCP-compliant client can use them. The animation shows the three-way handshake between client, server and tool.

← {contents:"..."}

🛠 Underlying Tools DBs · APIs · files · code

The MCP protocol in one paragraph

The client (an agent framework) asks the server "what tools do you have?" via tools/list. The server returns a schema. The client picks a tool and calls it via tools/call with arguments. The server executes it and returns the result. Same shape works for resources (files, documents) and prompts (reusable templates). Everything is JSON-RPC 2.0 over stdio or HTTP.

  • Uniform — every MCP tool looks identical to the agent, regardless of what it's wrapping.
  • Discoverable — agents can enumerate tools at runtime instead of being hard-coded to specific APIs.
  • Composable — stack multiple MCP servers; the agent gets the union of their tools.

Why MCP matters for enterprise security

MCP turns tool integration from a bespoke engineering problem into a governed interface — which means it can be secured at the protocol level instead of per-integration.

  • Audit — every tool call is a structured message you can log and replay.
  • Least privilege — scope tools per agent via the MCP server, not per client.
  • Supply-chain risk — pulling in a malicious MCP server is like pulling in a malicious npm package. OWASP MCP Top 10 covers this directly.
  • Real-world risk — the Postmark MCP server incident (Sept 2025) showed how a trusted MCP package can be updated to silently exfiltrate data.
Learn & Explain / Retrieval-Augmented Generation (RAG)
Concept · RAG

RAG — grounding LLM answers in your own data

Retrieval-Augmented Generation is the pattern that fixes an LLM's two biggest problems in enterprise: it doesn't know your data, and it hallucinates. RAG takes the user's question, retrieves the most relevant chunks from your own documents using vector search, and feeds them to the LLM as context. The 5 steps, demystified

  • Query — user asks a question in plain English.
  • Embed — an embedding model (e.g. text-embedding-3-small) converts the query into a high-dimensional vector that captures its meaning.
  • Retrieve — the vector is compared against your pre-embedded document chunks in a vector store (AI Search, Qdrant, Pinecone). Closest-k chunks come back.
  • Augment — those chunks are stuffed into the prompt as context, usually with phrasing like "Use only the following information to answer…"
  • Generate — the LLM answers the user's question using that retrieved context, with citations back to source chunks.

Common RAG failure modes & security risks

OWASP RAG-01 through RAG-06 catalogues the specific ways RAG pipelines break or get attacked:

  • RAG-01 Poisoned data — attacker plants malicious content in the ingest pipeline so retrieval returns it.
  • RAG-02 Injection via context — retrieved chunks contain prompt-injection payloads that hijack the LLM.
  • RAG-03 Leakage — sensitive data in chunks gets echoed to unauthorised users.
  • RAG-04 Access control bypass — vector store doesn't enforce per-user filters, so one user retrieves another's documents.
  • RAG-05 Low-quality retrieval — wrong chunks come back, LLM answers confidently with the wrong facts.
  • RAG-06 No observability — nobody logs which chunks were retrieved for which query, making debugging and forensics impossible.
Learn & Explain / Large Language Models (LLM)
Concept · LLM

LLM — tokens in, tokens out, attention in the middle

At heart, a Large Language Model is a next-token predictor. You feed it a sequence of tokens (words, subwords, punctuation), its attention layers weigh which input tokens matter for the current step, and it outputs one token at a time — each appended back into the context to predict the next. The four things every LLM does

  • Tokenise — break the input into subword units (roughly 4 characters each).
  • Embed — convert each token into a vector; positional encodings add "where in the sequence" info.
  • Attend — at each transformer layer, every token "looks at" every other token and decides what matters. This is the secret sauce — it's how the model knows that in "The bank of the river", bank means something different than in "I went to the bank".
  • Predict — the final layer outputs a probability distribution over all possible next tokens. Sampling strategies (temperature, top-p) pick one. Repeat until done.

Common LLM failure modes & security risks

OWASP LLM Top 10 names the big ones. All of these apply to models running in Microsoft Foundry just as they apply to OpenAI's direct API or open-weight models:

  • LLM01 Prompt injection — attacker crafts input that hijacks the model's instructions.
  • LLM02 Insecure output handling — the model's output is trusted as safe HTML / SQL / shell command.
  • LLM03 Training data poisoning — attacker contributes poisoned training data upstream.
  • LLM06 Sensitive information disclosure — model regurgitates PII or secrets it was trained on.
  • LLM07 Insecure plugin/tool design — function-calling tools have weak input validation.
  • LLM09 Overreliance — humans treat confidently-worded outputs as verified facts.
  • LLM10 Model theft — attacker extracts or exfiltrates the model weights through repeated API queries.
